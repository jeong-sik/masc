Adversarial Findings

Prioritized work items. P0 items are correctness or trust blockers.

todo

P0 Live dashboard is stale relative to inspected main

Live runtime reports source snapshot 76d88e2bc0, while repo head is 2eed0aa224. The latest schedule dashboard and keeper next-action work cannot be assumed deployed.

  • Rebuild/restart from current main.
  • Re-check /api/v1/dashboard/tools.
  • Require matching runtime/source identity or known binary commit before calling the dashboard current.

P0 Prompt suggests schedule tools that are not exposed

Keeper prompt/read-model can recommend masc_schedule_get, masc_schedule_approve, and masc_schedule_reject, but live inventory reports every schedule tool with no active surface.

  • Decide the intended schedule tool surface: operator-only, keeper-internal, or both.
  • Add all six masc_schedule_* tools to the proper catalog/surface.
  • Add regression coverage for descriptor registration and active exposure.

P0 Existing live schedules use unsupported payload kinds

The only concrete consumer supports masc.board_post. Live records include orphan_auto_release, keeper_surface_post, task_goal_generation, and backlog_depletion_check.

  • Add a consumer registry that declares supported payload kinds.
  • Implement or retire the currently observed payload kinds.
  • Surface unsupported payload kind counts and remediation in the dashboard.

P1 Dashboard exposure is buried in Tools/Lab

The full panel is useful for investigation, but it is weak as system monitoring. Operators can miss blocked approvals or failed schedules unless they open the tool inventory page.

  • Add a compact scheduled automation card to the overview/runtime/attention area.
  • Show FSM state, next due, blocked approval count, due-ready count, and unsupported failures.
  • Link to the full Tools/Lab table.

P1 UI is read-only where operators need decisions

The panel renders readiness and approval policy, but does not let the operator inspect, approve, reject, cancel, or recreate from the dashboard.

  • Add action affordances for get, approve, reject, cancel, and recreate.
  • Gate side-effecting actions behind the existing human grant model.
  • Render the result back into the row status or last-execution area.

P1 Runtime parser coverage is thin

The TypeScript type and component fixture exist, but there is no strong runtime parser or schema drift guard around scheduled_automation.

  • Add dashboard API normalization for the projection.
  • Totalize optional fields and warn on missing required fields.
  • Fixture active, blocked approval, unsupported failure, and stale projection cases.

P1 No live-safe happy-path smoke exists

Current live state proves terminal and unsupported cases, not due approved supported schedule to successful execution record.

  • Add a safe no-op payload kind or live-safe board-post smoke fixture.
  • Capture before/after dashboard projection evidence.

P2 Runner liveness is not first-class

The panel derives schedule state, but does not show last runner tick, last dispatch attempt, or stale-runner warnings.

  • Add last tick time, last tick error, and emitted/dispatch/rescheduled counters.
  • Surface stale-runner state in the overview card and full panel.

P2 Recurrence is under-explained operationally

The model supports recurrence and fixed-offset timezone contracts, but the UI mainly displays summaries.

  • Add previous-run, next-run, and recurrence explanation fields.
  • Test daily fixed-offset behavior around date boundaries.
