## 1. Purpose

The repo has **84 manual `.tla` specs** across 17 domains (per `specs/INDEX.md`, auto-generated by `scripts/gen-tla-index.sh`). The CLAUDE.md `TLA+ Bug Model 패턴` rule states:

> **양쪽 cfg 모두 통과해야 spec이 유효하다**: clean이 pass하는데 buggy도 pass하면 invariant가 너무 약한 것. clean이 fail하면 모델이 잘못된 것.

This audit measures *how widely the rule is applied* and surfaces three classes of gap. As with the OAS audit chain (Phase 1→3), this is a **survey, not a design proposal** — fixes need domain expertise per spec and are deferred.

## 2. Method

```bash
# Domain-level coverage
for dir in specs/*/; do
  domain=$(basename "$dir")
  tla=$(find "$dir" -maxdepth 2 -name "*.tla" 2>/dev/null | wc -l | tr -d ' ')
  cfg=$(find "$dir" -maxdepth 2 -name "*.cfg" 2>/dev/null | wc -l | tr -d ' ')
  buggy=$(find "$dir" -maxdepth 2 -name "*buggy*" 2>/dev/null | wc -l | tr -d ' ')
  printf "%-30s tla=%3d cfg=%3d buggy=%2d\n" "$domain" "$tla" "$cfg" "$buggy"
done

# Tautology Bug Models (clean == buggy invariant set)
# Inspect specs/INDEX.md columns clean={...} buggy={...} for textual equality
```

`specs/INDEX.md` is auto-generated and lists per-cfg invariants. Comparing the `clean=` and `buggy=` columns directly surfaces tautology pairs without any new tooling.

This **may** still be valid: the buggy cfg can override the `Spec`/`Init`/`Next` action (e.g. via a separate `SpecBuggy` definition that includes `BugAction`) while keeping the same invariants — that *is* the CLAUDE.md pattern. To confirm, the buggy `.cfg` file must reference a different `SPECIFICATION` than the clean `.cfg`.

```bash
# Sanity check (one example):
diff <(grep -E "^SPECIFICATION|^INVARIANT" specs/auth/AuthIdentityFSM.cfg) \
     <(grep -E "^SPECIFICATION|^INVARIANT" specs/auth/AuthIdentityFSM-buggy.cfg)
```

If the SPECIFICATION line is identical in both, the pair is a **true tautology** — buggy cfg adds no signal. Spot-checking a sample of the 21 candidates is the proper next step (deferred to a follow-up audit; needs domain-by-domain inspection).

The point is structural: `INDEX.md` reveals 21 candidate-tautology rows that warrant per-spec inspection, not that all 21 are wrong.

### 3.3 Split-pair Bug Models (separate `Bug.tla` file instead of unified spec)

`specs/bug-models/` contains the older split-pair pattern:

```
AmbiguousPartialCommit.tla     (clean only — buggy=0)
AmbiguousPartialCommitBug.tla  (buggy only — clean=0)
KeeperWorkPipelineBug.tla      (buggy only)
```

CLAUDE.md's pattern is **one spec file with `Spec` and `SpecBuggy` definitions** sharing the same module — `KeeperOASAdvanced.tla` is the canonical example (one file, `BugAction = CancelledAbsorbed`, invariant `CancelledNeverAbsorbed`). The split-pair style works, but:

- `INDEX.md` shows it as orphans (clean=0 or buggy=0 row) which the metric in §3.1 misclassifies.
- Two files mean the buggy variant can drift from the clean module's actual `Next` action without the divergence being visible.

The 3 split-pair specs in `bug-models/` predate the unified pattern and could be migrated; this is a hygiene improvement, not a correctness gap.

## 5. Why fixes are deferred

Each Bug Model needs a domain-specific `BugAction` describing a real production bug:
- `autonomous/AutonomousLoop`: a plausible bug is "tick fires while phase=Idle" — but the clean spec already has `TickOnlyDuringRunning`. The BugAction must be subtle enough that the invariant catches it without rejecting valid behavior.
- `multimodal/MultimodalArtifact`: domain expertise needed — what failure mode in the artifact lifecycle is realistic to model?

Writing a buggy cfg without that domain knowledge produces *fake* Bug Models — an `assert false` in the BugAction would trip any invariant. The audit's role is to *flag the gap*, not to fabricate models.

This mirrors the OAS audit chain's discipline: Phase 2 refined Phase 1's verdict via taxonomy (C1–C4), it didn't refactor Layer C blindly.

## 6. Recommended ratchet (descriptive, not enforced)

Following the pattern from `scripts/oas-boundary-ratchet.sh` (PR #12117 MERGED), a TLA+ Bug Model coverage ratchet would track:

```bash
# Strict (descriptive only — would be enforced after baseline review)
domains_without_bug_model: count of domains where buggy_cfg_count == 0 AND tla_count > 0
# Floor: 5 (current). Goal: monotonically decrease.

# Tautology surface (descriptive)
candidate_tautology_specs: count of specs where INDEX.md shows clean={X} buggy={X} textually
# Floor: 21 (current). Inform per-spec triage.
```

The ratchet wouldn't gate CI immediately — both metrics need a sweep PR (or several) before the floor becomes a meaningful enforcement target. The OAS chain spent four phases moving from "needs sweep" to "ratchet enforced" exactly for this reason.
