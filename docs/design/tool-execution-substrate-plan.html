Program Goal

The program goal is Keeper Tools Success: keepers should choose, execute, recover from, and explain tool calls without falling into legacy names, over-specific GitHub tools, raw shell ambiguity, or unobservable policy failures.

Goal ID proposal: goal-keeper-tool-execution-substrate-success-20260527

This goal is complete only when active keeper tools succeed through the typed substrate in realistic work loops, not merely when a descriptor enum compiles.

Objective Success metric Target Primary proof
Tool naming success Legacy or hallucinated public tool calls such as Read, Bash, dedicated PR wrappers, and gh-specific wrappers. 0 active-surface hits Descriptor projection tests plus active prompt/config/source ratchets.
Execute substrate success Command workflows represented as typed executable plus argv, then lowered to decided Shell IR. 100% for git/gh/dune test scenarios Keeper tool harness scenarios and Shell IR audit.
Recovery success Deterministic rejection returns the correct next tool or argument shape. No repeated same-args retries Workflow rejection tests and live log sampling.
Evidence success Every call records route, policy decision, decision reason, sandbox/backend, and Shell IR risk when applicable. Per-call JSONL is authoritative Receipt tests and dashboard projection checks.
Operator success A failed tool call can be diagnosed from one receipt row without reading three subsystems. Single-row diagnosis Tool call evidence and keeper detail dashboard.

Core Thesis

A tool should be a stable semantic capability, not a wrapper around a common command. GitHub comments, PR reviews, PR closes, and commits are command workflows. They belong inside Execute as typed gh or git argv. MASC-owned state transitions such as tasks, goals, board posts, approvals, memory, and keeper lifecycle remain structured tools.

Rejected failure mode: Dedicated PR comment/review/close wrappers and gh-specific PR/commit wrappers increased schema, policy, docs, prompt, receipt, and test coupling while adding no stronger safety boundary than typed Execute can provide.

Architecture

The descriptor is the contract. The executor is a primitive route. The sandbox is containment. Hooks observe. Receipts prove the decision.

Public Tool Model-facing name such as Execute or ReadFile.
Descriptor Schema, public name, internal name, policy, executor, backend, sandbox.
Typed Input Strict JSON. No raw shell strings for Execute.
Shell IR Typed command representation with decided risk envelope.
Gate Risk, write access, allowlist, path scope, approval.
Runtime Sandbox or host process dispatch through selected backend.
Receipt Route, policy decision, reason, risk, output, retry class.

Comparative Research

The external systems are useful as contrast, not as templates. MASC should borrow their safety lessons while rejecting tool-surface sprawl.

System Observed tool model Strength to borrow Risk to reject MASC stance
OpenClaw Small built-in tool set, optional skills/MCP, high-risk shell command tool disabled by default. Privilege shell execution and make built-ins understandable. Raw shell command as the normal execution language. Typed Execute only; raw shell does not become the model contract.
Hermes Agents Broad tools and toolsets, terminal backends, file/browser/memory/delegation tools, dynamic MCP tools. Toolsets and backend selection are useful for exposure control. Tool count becoming a second product API. Keep toolsets for capability grouping, but preserve primitive executor classes.
Claude Code local tree Many concrete tools plus central permission decisions and Bash guidance to prefer dedicated file/search tools. Structured file/search tools improve review, permissions, summaries, and diffs. Prompt guidance alone cannot hold a boundary under autonomous loops. Enforce boundaries in descriptors, tests, ratchets, and receipts.

Keeper Tools Success Harness

The harness should exercise the actual behavior that matters in production: tool selection, deterministic failure, recovery, and evidence quality.

Scenario Expected tool path Success condition Failure that must be caught

Admission Bar

New tools must pass this review before implementation. A "no" here should redirect the work to a skill, runbook, prompt, or typed Execute example.

Question Reject if yes Admit if true
Can this be done by Execute with typed argv? Do not add tool Only admit if the CLI shape cannot carry required review semantics.
Can this be done by file/search tools? Do not add tool Only admit if state is not file-system state.
Is this MASC-owned state? No, if it is vendor CLI state. Yes for tasks, goals, board, approvals, memory, keeper lifecycle.
Does it add a stable semantic contract? No, if it just wraps a vendor command. Yes, if it provides transactionality, typed receipts, or review UX.
Can receipt evidence explain the decision? No, if the decision hides in hooks or logs only. Yes, if per-call JSONL records decision source and reason.

Risk Register

Risk Impact Mitigation Owner surface
Convenience pressure reintroduces GitHub micro-tools. Tool sprawl, prompt drift, test bloat, vendor coupling. Active-surface ratchet and admission checklist. Descriptor + prompts + tests.

Static descriptor evidence lacks actual policy decision. Operators cannot diagnose why a call failed. Add per-call policy_decision, source, reason, risk class. Receipt and dashboard.
Prompt names diverge from active schema names. Keepers call missing tools and waste turns. Prompt/matrix alignment tests and hard-cut surface checks. Prompt registry and tool alias tests.
Transient infra contention is reported as workflow rejection. Keepers stop or self-correct when they should retry after lock pressure drops. Keep lock contention typed through Claim_next_transient_error and return transient tool failures with holder diagnostics. Task claim runtime and tool result tests.
Hooks become hidden policy owners. Policy behavior becomes hard to reason about and hard to test. Hooks observe only; policy stays in descriptors/gates. Hook observation boundary tests.

Research-Style Evaluation Plan

Treat the tool substrate as a small empirical system. Each experiment should produce durable JSON evidence, not a free-text success claim.

Hypothesis H1
A seven-tool public surface plus MASC-owned in-process domain tools reduces tool-name errors compared with legacy aliases and micro-tools.
Measure: unknown tool calls, legacy-name calls, and correction-loop count per 100 keeper turns.
Hypothesis H2
Typed Execute through Shell IR is safer than raw shell strings for git/gh/dune workflows without needing GitHub-specific tools.
Measure: deterministic reject quality, same-args retries, successful gh/git scenarios, and Shell IR risk coverage.
Hypothesis H3
Per-call decision evidence reduces operator diagnosis time for tool failures.
Measure: number of files/log streams required to explain a failure before and after receipt enrichment.
Hypothesis H4
Prompt/schema alignment prevents Tool not found: Read and similar wake-loop failures.
Measure: stale prompt name scan and live failed-tool logs.
