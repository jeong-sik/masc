Keeper Failure Policy Layer Plan

Keeper를 죽일 실패와 턴만 실패시킬 실패를 한 계층에서 판정하도록 만드는 실행 계획서. 목표는 임시 예외처리나 문자열 기반 우회가 아니라, failure taxonomy, lifecycle disposition, operator visibility를 하나의 typed decision으로 연결하는 것이다.

Executive Goal

현재 문제는 keeper 실패가 여러 callsite에서 서로 다른 언어로 해석된다는 점이다. keeper_bash workflow rejection, oas_timeout_budget, stale watchdog, supervisor auto-pause, dashboard blocked count가 서로 다른 정책 계층처럼 동작하면서 operator에게는 모두 "keeper가 막힘/죽음"으로 보인다.

최종 상태는 모든 실패가 먼저 Keeper_failure_policy.decide를 거쳐 disposition을 받고, heartbeat/supervisor/watchdog/dashboard는 그 결과만 소비하는 구조다. 

failure evidence
  -> Keeper_failure_policy.decide
  -> { failure_class; scope; disposition; lifecycle_effect; operator_action; evidence }
  -> runtime action + metrics/logs + dashboard status

Current Problem Statement

Observed Confusion

  • Paused, Blocked, Backoff, Provider cooldown이 UI/로그에서 섞인다.
  • blocked_count가 실제 blocked task 수가 아니라 fleet capacity shortfall로 보인다.
  • Oas_timeout_budget_loop가 provider/turn 증상인데 keeper lifecycle event로 승격된다.
  • stale watchdog은 root cause를 보존하려다 lifecycle action까지 과하게 이어질 수 있다.

Policy Model

1. Evidence typed error, provider health, skip reasons, watchdog class, tool result, committed tools, slot holder age
2. Scope provider/call, turn, keeper fiber, durable keeper, fleet/system
3. Disposition feedback, retry, soft_backoff, route_away, restart, pause, dead, operator_required
4. Visibility metric labels, structured log, dashboard status, next operator action

Proposed Decision Type

type failure_scope =
  | Provider_call
  | Keeper_turn
  | Keeper_fiber
  | Keeper_durable_state
  | Fleet_system

type lifecycle_effect =
  | No_lifecycle_effect
  | Turn_soft_fail
  | Provider_cooldown
  | Keeper_restart
  | Keeper_auto_pause
  | Keeper_operator_pause_required
  | Keeper_dead

type operator_action =
  | No_action
  | Inspect_timeout_budget
  | Inspect_provider_health
  | Fix_cascade_config
  | Fix_tool_contract
  | Fix_runtime_environment
  | Resume_after_backoff

type decision = {
  failure_class : string;
  scope : failure_scope;
  disposition : string;
  lifecycle_effect : lifecycle_effect;
  operator_action : operator_action;
  breaker_applicable : bool;
  auto_resume : bool;
  evidence : Yojson.Safe.t;
}

Failure Taxonomy

Failure class Default scope Lifecycle effect Escalates only when Operator wording
workflow_rejection turn/tool feedback none never to keeper death; repeated misuse may nudge instructions Command rejected by workflow guard. Rewrite tool usage.
provider_timeout / oas_timeout_budget provider call or keeper turn turn soft fail only with separate liveness evidence such as wedged child process or leaked slot Provider or turn budget exhausted. Inspect timeout budget and provider health.
transient_network / overload provider call retry/backoff sustained provider-wide failure fingerprints across cooldown windows Transient upstream issue. Route away or retry after backoff.
cascade_config_exhausted cascade policy operator-visible pause possible all configured candidates missing auth, unavailable, or filtered with no fallback Cascade has no callable providers. Fix config or credentials.
required_tool_contract turn/tool contract pause current work possible required tool is impossible for the active keeper/provider surface Required tool contract cannot be satisfied. Adjust provider/tool policy.
fatal_environment keeper fiber/runtime restart or pause environment invariant broken: missing Eio context, sandbox corruption, FD exhaustion Runtime environment is unhealthy. Fix host/runtime before resuming.
stale_in_turn_hung keeper fiber force-release + restart active turn exceeds hard liveness budget or no-progress threshold Keeper turn is hung. Restart fiber and preserve turn evidence.
stale_storm fleet/system durable pause possible same keeper or cohort repeatedly restarts into stale termination Repeated liveness storm. Stop churn and require operator inspection.

OAS Timeout vs Hopeless

Timeout, but not hopeless

  • single provider timed out while fallback providers remain callable
  • timeout phase is stream_idle with answer/thinking/tool-call progress already observed
  • provider health has recent success or short cooldown only
  • prompt/context was unusually large for this turn
  • outer keeper turn remains cancellable and slots are released
  • no hard quota, auth, model-not-found, or required-tool terminal fingerprint

Potentially hopeless without operator action

  • all cascade candidates are missing credentials, hard-quotaed, or model-not-found
  • required tool contract cannot be satisfied by any selected provider
  • timeout phase is provider_step and all fallback candidates are exhausted or disabled
  • same terminal fingerprint repeats after cooldown windows
  • active child process or slot is wedged and force-release evidence exists
  • the same keeper enters a stale termination storm after restart
Policy rule: oas_timeout_budget alone may set Turn_soft_fail or Provider_cooldown. It may not set Keeper_auto_pause or Keeper_dead unless a second typed evidence source proves keeper/runtime liveness is broken.

Implementation Sequence

Phase 1: Pure Classification

  • Create closed decision vocabulary.
  • Map existing typed errors into decisions without side effects.
  • Add table-driven tests before callsite migration.

Phase 2: Runtime Migration

  • Replace direct raise Keeper_fiber_crash where failure is not fatal.
  • Replace supervisor auto-pause case splits with policy decisions.
  • Keep fatal environment and true stale-hung restart paths strict.

Phase 3: Operator Clarity

  • Rename capacity surfaces away from ambiguous blocked wording.
  • Expose next action and lifecycle effect in dashboard/runtime trust.
  • Backfill docs and runbooks with new vocabulary.

Code Migration Targets

Target Current responsibility Desired dependency direction
keeper_heartbeat_loop.ml Detects OAS timeout and currently owns parts of strike handling. Build evidence, call policy, apply returned turn/lifecycle action.
keeper_supervisor.ml Maps crash reasons to restart, pause, dead, and auto-resume. Consume lifecycle_effect; stop duplicating failure taxonomy.
keeper_stale_watchdog.ml Classifies stale kills and preserves some prior failure reasons. Produce liveness evidence; policy decides whether prior cause affects lifecycle.
keeper_error_classify.ml Classifies recoverability and retry semantics. Remain error-level classifier; policy composes it with liveness and provider evidence.
keeper_status_bridge.ml Projects runtime blockers into dashboard status. Render policy decisions and next action without conflating pause/block/backoff.

Acceptance Criteria

  • A1 A provider/turn timeout cannot directly kill or durable-pause a keeper.
  • A2 Fatal runtime failures still trip restart/circuit breaker paths.
  • A3 Dashboard separates operator_paused, auto_backoff, provider_cooldown, capacity_shortfall, and true task blocked state.
  • A4 Every failure decision has a structured log/metric with failure_class, scope, disposition, and lifecycle_effect.
  • A5 Regression tests prove the negative cases: workflow rejection does not break keeper; OAS timeout does not break keeper; fatal environment still breaks keeper.
  • A6 Runbooks no longer describe repeated OAS timeout as keeper auto-pause by default.
