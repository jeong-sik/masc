Keeper Tool Runtime Boundary Plan

Goal-driven cleanup for the confusing Keeper_shell, Keeper_bash, GitHub, Docker, and Local execution split.

Updated 2026-05-24 Branch refactor/sandbox-backend-actor-boundary-main Worktree .worktrees/sandbox-backend-actor-boundary

Decision

Backend is not a tool surface

Docker or Local sandbox selection must be orthogonal to command semantics. Git, GitHub, typed Bash, shell inspection, and PR workflows should lower into backend-neutral requests before the runtime chooses Docker or Local execution.

Current Baseline

0 Old bad helper names in lib/keeper
0 Direct Docker runner calls outside Keeper_sandbox_runner
0 Backend host exec calls using Keeper_shell actor

Slice 1 removes source-level references to Keeper_shell_docker, Keeper_shell_docker_exec_failure, and Keeper_shell_bash_docker. Slice 2 removes the legacy Keeper_shell_shared.run_docker* boundary and routes PR/GitHub tool execution through Keeper_sandbox_runner. Slice 5 removes the legacy keeper_shell op=gh and op=git_clone compatibility surface instead of preserving bridge shims. Slice 6 moves shell read-side sandbox execution behind Keeper_sandbox_read_runner. Slice 7 applies the same boundary to file tools. Slice 8 retires the tool-shaped Keeper_docker_read module name. Slice 9 moves backend host process execution to the sandbox backend actor. Runner tests use injected mock runners.
