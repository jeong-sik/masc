고객 유용성 결론: 개발자·운영자가 제한된 canary를 시작하고 새 요청 일부를 처리할 기반은 회복됐다. 그러나 일반 고객에게 “맡기고 떠나도 결과가 온다”고 약속할 수 있는 제품은 아니다. 608개 pending과 2개 inflight를 정상 처리·격리·실패 종결하고 target을 10으로 복구하는 증거 없이 채널 수나 Keeper 수를 늘리면 고객 가치는 늘지 않고 지연과 불신만 증폭된다.

9.0 release floor: 위 6개 축의 가중 합 ≥ 9.0이고, 동시에 실행 가능성·terminal exactly-once·silent loss 세 hard gate가 모두 통과해야 한다. 평균 점수가 높아도 이 셋 중 하나가 0이면 출시 불가다.

Crash matrix: candidate commit 전 crash는 작업 미접수로 끝나고, commit 후 wake 전 crash는 startup scan이 복구한다. Running 중 process crash는 다음 worker epoch가 Ready로 회수하고, process가 살아 있는 상태의 독립 lane exception/cancellation은 동일 worker epoch의 claim만 Ready로 해제한다. 이미 Completed/Deferred/Blocked로 commit된 partition은 해제하지 않으며 다른 epoch의 claim도 침범하지 않는다. Completed 후 candidate apply 전 crash는 Completed replay, event enqueue 후 Consumed commit 전 crash는 candidate ID 기반 enqueue replay, Consumed 후 Settled 전 crash는 exact judgment replay 후 Settled로 수렴한다. 서로 다른 durable file을 하나의 atomic transaction으로 가장하지 않는다.

현재 검증 경계: strict removed-state rejection, response failure no-split, exact settlement replay, owner nonblocking, sibling cancellation isolation, domain-safe wake stream, per-ledger startup recovery, cross-Keeper path collision, typed health projection을 focused target과 CI에서 검증한다. Candidate/partition JSONL parse·rewrite와 provider-free owner delivery는 systhread로 분리하되 OAS judge는 Eio lane fiber에 남긴다. 이 구현 단위 증거는 8-hour fleet와 direct-latency SLO를 대신하지 않는다.

5. TO-BE architecture

Load-bearing invariant

Keeper owner admission에는 사용자 대화, 상태 결정, 짧은 durable commit만 들어간다. provider batch 판단, 이미지 생성, 장기 코드 실행, 리서치, 음성 합성은 job plane에서 실행하고 결과를 durable stimulus로 되돌린다.

6. No-heuristic mechanism

AFK is explicit, never guessed

type operator_mode =
  | Interactive
  | Catch_up of { operation_id : Operation_id.t }

type mode_source =
  | Dashboard_operator
  | Scheduled_operation
  | Api_operator
  • “마지막 키보드 입력 후 N분”으로 Away를 추정하지 않는다.
  • Catch-up 시작/종료는 durable operation으로 기록한다.
  • 어떤 mode에서도 direct queue가 존재하면 ambient worker는 owner admission을 얻지 않는다.
  • Catch-up은 provider permit을 늘릴 수 있지만 owner admission의 single-flight invariant는 깨지 않는다.

Capacity is observed through typed contracts

Ready → Running
  ├─ exact response IDs → Completed
  ├─ response/provider/runtime unavailable → Deferred
  ├─ membership/delivery invariant violation → Blocked
  └─ future typed actual-wire overflow → split authority not implemented yet
  • candidate 개수 8, bytes/4, prose error 문자열을 안전 근거로 사용하지 않는다.
  • partition membership와 요청/응답 ID 집합은 exact equality로 검증한다.
  • 일반 provider outage는 split하지 않는다. 그렇지 않으면 장애 한 번이 2N−1 호출 폭풍이 된다.
  • context overflow 자동 split은 OAS가 actual-wire capacity failure를 typed contract로 제공한 뒤에만 허용한다.

Typed work and state transitions

type work_class =
  | Direct_input of Keeper_continuation_channel.t
  | Approval_resolution

  | Background_completion of Job_id.t

  | Pending
  | Leased of Lease.t
  | Judged of Judgment.t
  | Delivery_blocked of Durable_error.t

  | Submitted
  | Running of Resource_lease.t
  | Completed of Artifact_ref.t

TTL로 임의 폐기하지 않는다. 같은 의미 슬롯의 새 signal이 도착하면 Superseded, Task가 종료되면 명시적 lifecycle transition, 운영자가 폐기하면 operation receipt를 남긴다.

PR-1 — Follow-up to merged #25073: fairness and correctness

  1. blind TTL/Expired 정책을 제거하고 typed defer/supersession 계획으로 전환한다.
  2. 동일 keeper_context fingerprint만 한 batch에 포함한다.
