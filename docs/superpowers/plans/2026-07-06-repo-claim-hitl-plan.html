Keeper Repository Claim HITL Plan

Unknown keeper repository access should not silently become authority, but it should also not dead-end as a plain reject when the repository identity is known and the only missing fact is operator authorization.

Date: 2026-07-06 Repo: jeong-sik/masc Boundary: MASC only OAS impact: none

Decision

Keep the strict repo access gate as the default authority. Add explicit HITL recovery paths for sandbox clones that are physically present but not authorized by the repository catalog. Registered repositories remain allowed because keeper mappings are advisory/default-scope metadata, not access caps.

Allowed

Existing mapping permits the repository, or the current default registered-repository scope applies.

HITL Pending

A sandbox clone has auditable Git metadata but is missing from repositories.toml, or its sandbox name should become an explicit alias of an existing catalog repository.

Denied

Repository store load failed, Git metadata is incomplete, or identity evidence is insufficient for an automatic approved mutation. The keeper remains fail-closed.

Implementation Slice

  • Keep Keeper_repo_mapping as the pure policy/enforcement boundary.
  • Add a keeper-side HITL bridge that calls Keeper_approval_queue.submit_pending for sandbox clone registration requests.
  • On operator approve, update repositories.toml through Repo_store.add for new repositories, or Repo_store.update for approved aliases.
  • On reject, edit, or incomplete evidence, persist no catalog mutation and emit an explicit log path.
  • Wire read/write filesystem paths to return a structured approval_pending response instead of a plain policy reject for claimable sandbox clones.

Non Goals

  • No OAS API or provider behavior change.
  • No automatic repo registration without an operator approval decision.
  • No duplicate catalog entry when the clone remote matches an existing registered repository; use an approved alias instead.
  • No heuristic matching from free text, branch names, or command strings.
  • No background mutation without an approval decision.

Validation

Case Expected Result Check
Unregistered sandbox clone with valid origin metadata HITL pending approval is created; approve registers the repo in repositories.toml Focused Alcotest for repo registration bridge
Clone name differs but origin remote matches an existing repo HITL pending approval is created; approve adds the clone segment as an alias Resolve pending approval and reload repository catalog
Unregistered repo id without sandbox clone evidence Fail closed; no HITL request Repo access decision test
Malformed mapping or repository store Fail closed with explicit diagnostic Existing fail-closed tests remain unchanged
