Keeper Tool Runtime Boundary Plan

Goal-driven cleanup for the confusing Keeper_shell, Keeper_bash, GitHub, Docker, and Local execution split.

Updated 2026-05-24 Branch refactor/keeper-sandbox-runner-facade Worktree .worktrees/keeper-sandbox-runner-facade

Decision

Backend is not a tool surface

Docker or Local sandbox selection must be orthogonal to command semantics. Git, GitHub, typed Bash, shell inspection, and PR workflows should lower into backend-neutral requests before the runtime chooses Docker or Local execution.

Current Baseline

0 Old bad helper names in lib/keeper
0 Direct Docker runner calls outside Keeper_sandbox_runner
3,706 Combined LOC across main hot files and new sandbox helper modules

Slice 1 removes source-level references to Keeper_shell_docker, Keeper_shell_docker_exec_failure, and Keeper_shell_bash_docker. Slice 2 removes the legacy Keeper_shell_shared.run_docker* boundary and routes PR/GitHub tool execution through Keeper_sandbox_runner.

Goals

ID Goal Quantitative Gate Status
G1 Remove tool-surface names from sandbox backend helper modules. rg source hit count for retired helper modules is 0 under lib/keeper. Done in slice 1
G2 Introduce a backend-neutral command runner facade so tool modules stop calling Docker helpers directly. Direct Keeper_sandbox_docker.run_* and Keeper_shell_shared.run_docker* call count in non-test source reaches 0 outside the facade. Done in slice 2
G3 Move Git/GitHub semantics out of keeper_shell_ops.ml toward GitHub domain modules and native PR tools. keeper_shell_ops.ml LOC drops below 1,200, and op=gh tests pass through compatibility coverage. Planned
G4 Make Local and Docker command failures share one semantic result shape. Golden tests show identical semantic fields for matched Local/Docker failures, with backend details only in optional evidence fields. Planned
