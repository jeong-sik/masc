설계서 — Keeper 턴 상세: thinking↔tool 인터리빙 보존

상태 Draft (설계 검토용) 작성 2026-06-21 출처 턴 상세 적대 감사 concern #5 연관 #21913(구현 PR), #21748 (thinking-trace mis-attribution), #21889(머지), #21892(머지) 대상 repo masc / dashboard

요약 (TL;DR)

키퍼 턴 상세에서 모델이 think → tool → think → tool 순으로 작업해도 화면에는 [생각 블록 전체][도구 블록 전체] 두 덩어리로만 보여, 어느 생각 다음에 어떤 도구가 실행됐는지를 운영자가 볼 수 없다. 원인은 (1) 라이브 스트림이 모든 tool entry를 assistant entry 앞으로 hoist하고, (2) ToolTraceCard가 thinking과 tool을 두 섹션으로 분리 렌더하기 때문이다. 핵심: 타입(ChatTraceToolStep)과 렌더러(ChatTraceStep 컴포넌트)는 인터리빙 표현력을 일부 갖고 있지만, 순수 kind:'tool' migration은 tool output join/status taxonomy를 잃는다. 현행 권장안은 #21913의 render-time occurrence merge — 단일 turn-local occurrenceSeq를 think/reason step과 tool entry 양쪽에 찍고 그 순서로 정렬해 #21892 missing/output 집계를 보존하는 방식이다.

2026-06-21 결정 보정

초기 설계는 옵션 C(도구를 traceSteps 안으로 이동)를 권장했지만, #21913 구현 리뷰에서 순수 옵션 C는 거부됐다. 이유는 ChatTraceToolStep에 tool_use_id가 없어 output store join을 끊고, status를 ok|err로 축소하며, #21892의 missing/aggregate 계산을 깨기 때문이다. 이 문서는 #21913의 render-time merge 결정을 SSOT로 따른다.

1. 문제 정의 (concern #5)

사용자 관찰: "Thinking 중간에 Tools가 나오는 배치는 못 보는건가? 이게 어떻게 묶이는건지."

턴 상세 transcript는 한 턴의 작업 과정을 "작업 과정" 카드(ToolTraceCard)로 보여준다. 그러나 모델의 실제 작업 순서(생각과 도구 호출의 교차)가 렌더링에서 소실되어, 운영자는 다음을 판단할 수 없다:

  • 특정 도구가 어느 추론 단계 직후에 실행됐는가
  • 도구 결과가 그 다음 생각에 반영됐는가(인과 순서)
  • 생각만 길게 하고 도구는 마지막에 몰아친 것인지, 생각-도구를 번갈아 한 것인지

이 정보는 키퍼가 "생각만 맴돌다 멈추는"(thinking-only/no-progress) 패턴을 운영자가 진단할 때 직접적 단서가 된다. concern #3(thinking 루프)과 같은 뿌리의 관측성 문제다.

2. As-Is — 현재 동작

대상 표면은 groupToolCalls=true 경로(keeper-shared.ts)이며 buildChatRenderUnits → TurnWorkBundle → ToolTraceCard로 렌더된다.

2.3 정보 손실 지점

순서 정보는 존재하지만 두 곳에서 폐기된다

  • (a) 스트림 hoist — keeper-stream.ts:81: tool entry를 assistant 앞으로 삽입하는 순간, 그 도구가 "어느 thinking 세그먼트 다음"이었는지가 사라진다. tool의 timestamp는 남지만 thinking 델타에는 세그먼트별 timestamp가 없다.
  • (b) 2-섹션 렌더 — ToolTraceCard가 traceSteps(생각)와 tools(도구)를 별도 입력으로 받아 순서대로 합치지 않고 두 섹션으로 그린다.

3. 핵심 통찰 — 표현력은 이미 있다

"새 기능"이 아니라 "미배선"

ChatTraceStep 유니온은 이미 tool 변형을 포함한다:

// types/core.ts:825–826
export type ChatTraceToolStep = { kind: 'tool'; name: string; status?: 'ok'|'err'; dur?: string; args?: unknown; result?: string }
export type ChatTraceStep = ChatTraceThinkStep | ChatTraceReasonStep | ChatTraceToolStep

그리고 ChatTraceStep 렌더 컴포넌트(primitives.ts)는 kind:'tool' 스텝을 이미 그릴 수 있다. 다만 이 타입은 tool output join/status SSOT가 아니므로 순수 migration 대상은 아니다. 핵심은 렌더러 표현력은 일부 이미 있지만, 현재 data path는 occurrence ordering key를 공유하지 않는다는 점이다.

도구 호출은 output join/status/missing 집계의 SSOT인 sibling entry로 유지한다. 대신 live stream 발생 시점에 thinking step과 tool entry 양쪽에 같은 turn-local occurrenceSeq를 stamp하고, ToolTraceCard가 렌더 직전에 두 입력을 sequence로 합성한다. 이 방식은 tool entry 기반 집계를 보존하면서 화면 순서만 실제 발생순으로 복원한다.

5. 설계 옵션 비교

옵션메커니즘변경 레이어침습도트레이드오프
A. 카드 occurrence merge (#21913) ToolTraceCard에서 traceSteps+tools를 turn-local occurrenceSeq로 정렬해 단일 .map. think/reason step과 tool entry는 같은 카운터에서 발급된 sequence를 가진다. 렌더만 (primitives.ts) 낮음~중간 채택 — 기존 tool entry를 유지해 output join, missing 집계, status taxonomy를 보존한다. 단, wall-clock timestamp가 아니라 스트림 발생 순서를 나타내는 단조 증가 sequence를 stamp해야 하며 sequence 없는 backend-normalized step은 legacy two-section fallback으로 둔다.
B. thinkAnchor 기록 keeper-stream.ts:81에서 tool 삽입 시 현재 assistantEntry.traceSteps.length(또는 thinking-char offset)를 tool entry에 thinkAnchor로 저장. 카드가 그 anchor 뒤에 도구 배치 stream + state + 렌더 중간 기존 entry 구조 유지하며 위치 재구성. anchor 계산 정확도에 의존. tool은 여전히 sibling entry라 두 자료구조를 동기화해야 함.
C. 통합 traceSteps tool 발생 시 assistant의 traceSteps에 kind:'tool' 스텝을 순서대로 append. TOOL_CALL_END에서 status/dur 후속 패치. 카드는 단일 배열 순회 stream + state (+ 카드 소폭) 중간 순수 migration은 기각 — ChatTraceToolStep에는 output-store join에 필요한 식별자와 현재 status taxonomy가 부족하다. 이를 보완하려면 사실상 ToolCallEntry를 중복하는 구조 변경이 필요하다.
D. timestamp 표면화 각 스텝에 상대 시각/순번 배지 표시 (A~C와 병행 가능) 렌더만 낮음 보조책. 순서를 복원하진 않지만 운영자가 순번을 추론하게 함. thinking 델타 timestamp 부재가 한계.

6. 현행 권장안 + 근거

옵션 A — #21913 render-time occurrence merge

  1. 기존 tool entry를 보존: lookupToolCallOutput(entry.id), semantic status, missing aggregation이 그대로 유지된다.
  2. 순서 복원에 필요한 최소 데이터만 추가: live turn마다 단일 occurrenceSeq 카운터를 두고, think/reason 세그먼트 시작과 tool CALL_START에서 같은 카운터를 증가시킨 뒤 그 값으로 정렬한다.
  3. 카드 렌더만 단일 순회로 단순화: ToolTraceCard의 thinking/tools 2-섹션 분기를 occurrence-ordered 단일 리스트로 바꾼다.

옵션 C는 구조적으로 깔끔해 보이지만 현재 타입으로는 output join/status 손실이 크다. 장기적으로 tool step을 SSOT로 삼으려면 ToolCallEntry급 식별자와 상태 모델을 먼저 통합해야 한다.

8. 테스트 계획

  • 인터리빙 보존: [think A, tool X, think B, tool Y] 스트림 → 렌더 DOM 순서가 정확히 A·X·B·Y인지 단언.
  • timestamp tie 회귀: think step과 tool entry의 wall-clock timestamp가 같은 경우에도 occurrenceSeq가 A·X·B·Y 순서를 결정하는지 단언.
  • output join 보존: interleave 후에도 tool output/missing/status가 기존 ToolCallEntry 기반으로 유지되는지 확인.
  • 회귀: 기존 think-only 턴, tool-only 턴, flat(groupToolCalls=false) 경로가 그대로 동작.
  • #21892 연동: settled 턴의 미join tool 스텝이 통합 배열에서도 missing으로 표시되는지.

9. 리스크 · 미해결 질문

충돌 · 좌표

  • #21913 구현 SSOT: 현행 구현은 #21913이다. #21748("assistant thinking-trace mis-attribution across identical-text turns")은 같은 keeper-state.ts/keeper-stream.ts traceStep 경로를 다루므로, 후속 변경은 #21913 위에서 rebase/통합해야 한다.
  • 순수 옵션 C의 이중 자료구조: tool을 step으로 옮기면, tool을 sibling entry로 읽던 다른 소비자(툴 실행 추적 패널, 그룹핑)와의 동기화/이중화를 정리해야 한다 — 잘못하면 한쪽만 갱신되는 silent drift.
  • sequence 없는 legacy/history step: occurrenceSeq가 없는 기존/backend-normalized step은 legacy two-section fallback으로 남는다. 완전한 과거 재구성은 별도 backend sequence surface가 필요하다.
  • sequence SSOT drift: think/reason과 tool이 서로 다른 카운터나 서로 다른 clock을 쓰면 tie/clock-skew 문제가 재발한다. 카운터는 turn/entry 단위의 단일 ordered source여야 한다.
  • 스트림 재정렬 부작용: hoist 제거가 자동 스크롤/스트리밍 표시(streamState)와 상호작용할 수 있어 라이브 회귀 테스트 필요.
