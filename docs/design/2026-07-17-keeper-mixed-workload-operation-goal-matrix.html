고객 유용성 결론: 개발자·운영자가 제한된 canary를 시작할 기반은 회복됐지만, 일반 고객에게 “맡기고 떠나도 결과가 온다”고 약속할 수 있는 제품은 아니다. 1,130개 pending을 정상 처리·격리·실패 종결하는 증거 없이 채널 수나 Keeper 수를 늘리면 고객 가치는 늘지 않고 지연과 불신만 증폭된다.

9.0 release floor: 위 6개 축의 가중 합 ≥ 9.0이고, 동시에 실행 가능성·terminal exactly-once·silent loss 세 hard gate가 모두 통과해야 한다. 평균 점수가 높아도 이 셋 중 하나가 0이면 출시 불가다.

Semantic judgment가 owner admission 안에 있다. 느린 provider가 한 Keeper의 모든 입력·도구·메모리 진행을 막는다.

Keeper turn
   ├─ provider reasoning
   ├─ code / research / voice / image tool
   │      └─ inline wait이면 같은 turn 계속 점유

10 Keeper는 10개 lane 병렬성이 있지만 각 Keeper 내부는 single-flight다. 한 lane의 장기 작업을 background job으로 분리하지 않으면 해당 인격은 작업 완료까지 deaf 상태다.

5. TO-BE architecture

Load-bearing invariant

Keeper owner admission에는 사용자 대화, 상태 결정, 짧은 durable commit만 들어간다. provider batch 판단, 이미지 생성, 장기 코드 실행, 리서치, 음성 합성은 job plane에서 실행하고 결과를 durable stimulus로 되돌린다.

6. No-heuristic mechanism

AFK is explicit, never guessed

type operator_mode =
  | Interactive
  | Catch_up of { operation_id : Operation_id.t }

type mode_source =
  | Dashboard_operator
  | Scheduled_operation
  | Api_operator
  • “마지막 키보드 입력 후 N분”으로 Away를 추정하지 않는다.
  • Catch-up 시작/종료는 durable operation으로 기록한다.
  • 어떤 mode에서도 direct queue가 존재하면 ambient worker는 owner admission을 얻지 않는다.
  • Catch-up은 provider permit을 늘릴 수 있지만 owner admission의 single-flight invariant는 깨지 않는다.

Batch capacity is declared, not guessed

type batch_budget =
  { runtime_id : Runtime_id.t
  ; max_context_tokens : int
  ; max_output_tokens : int option
  ; tokenizer : Provider_tokenizer.t
  }

Token_accounting_unavailable
  ⇒ typed defer, never byte/4 fallback
  • model max-context는 Runtime/OAS capability SSOT에서 읽는다.
  • 정확한 provider tokenizer가 없으면 batch admission을 거부한다.
  • candidate 개수 8이나 bytes/token 환산을 안전 근거로 사용하지 않는다.
  • output schema의 요청 ID 완전 일치가 아니면 typed partial failure다.

Typed work and state transitions

type work_class =
  | Direct_input of Keeper_continuation_channel.t
  | Approval_resolution
  | Compaction_recovery
  | Background_completion of Job_id.t
  | Ambient_attention of Candidate_id.t

type attention_state =
  | Pending
  | Leased of Lease.t
  | Judged of Judgment.t
  | Delivery_blocked of Durable_error.t
  | Consumed of Receipt.t
  | Superseded of Candidate_id.t

type job_state =
  | Submitted
  | Running of Resource_lease.t
  | Completed of Artifact_ref.t
  | Failed of Job_error.t
  | Cancelled of Cancellation_receipt.t

TTL로 임의 폐기하지 않는다. 같은 의미 슬롯의 새 signal이 도착하면 Superseded, Task가 종료되면 명시적 lifecycle transition, 운영자가 폐기하면 operation receipt를 남긴다.

8. PR slicing and exact order

PR-0 — Restore executable truth

  1. v3 snapshot validator를 offline/live 동일 parser로 실행한다.
  2. 10개 목표 Keeper가 등록되고 executable인지 확인한다.
  3. queue pending/inflight counts가 complete인지 확인한다.
  4. Discord/Slack ingress smoke는 Keeper turn receipt까지 확인한다.

Exit: 10/10 executable, queue read errors 0, direct receipt 10/10. 이 단계 전에는 성능 실험 금지.

PR-1 — Follow-up to merged #25073: fairness and correctness

  1. blind TTL/Expired 정책을 제거하고 typed defer/supersession 계획으로 전환한다.
  2. 동일 keeper_context fingerprint만 한 batch에 포함한다.
  3. 요청 ID와 응답 ID의 완전성 검증, partial outcome을 typed failure로 기록한다.
  4. owner admission당 정확히 한 provider batch만 실행하고 remaining이면 durable continuation wake를 남긴다.
  5. 현재 890 fixture로 calls 112, admission 112, cross-context 0, silent drop 0을 검증한다.

Live state: #25073은 2026-07-17 16:31 KST에 merge됐다. 아래 수정은 merge된 TTL·batch 경로를 되돌리는 것이 아니라 최신 main 위 후속 draft PR로 제출한다.

Exit: Build/Test green + mixed context + direct preemption + 439/451 backlog simulation.

Exact-token dependency: OAS PR #2647은 2026-07-17에 merge되어 prepared completion request의 provider-owned input count와 output receipt를 제공한다. 현재 구현은 Anthropic만 지원하며, 측정한 request artifact와 실제 dispatch artifact의 동일성을 아직 보장하지 않는다고 인터페이스가 명시한다. 따라서 #25073의 batch_max_candidates = 8은 1.0 승인 기준이 아니며, OAS release/pin → 동일 artifact fit admission 연결 전까지 hardcoded-cap blocker로 취급한다. 즉 dependency merge는 완료됐지만 MASC 제품 경로는 아직 완료되지 않았다.

PR-2/3 — Atomic ledger and signal-centric routing

  1. batch verdict를 한 durable rewrite로 commit한다.
  2. Board audience를 Targets/Broadcast/ThreadParticipants/Discoverable closed sum으로 만든다.
  3. direct signal을 non-target Keeper candidate로 fan-out하지 않는다.
  4. Discoverable만 owner lane 밖의 attention router로 보낸다.

Exit: direct amplification equals explicit target count; discoverable amplification and cost are 별도 지표.

PR-4..7 — Background jobs, media, memory

  1. 기존 fusion spawn→wake pattern을 generic typed job state로 확장한다.
  2. code/research/image/voice를 capability와 resource lease로 admission한다.
  3. inner Eio switch로 FD를 job 종료 시 회수한다.
  4. 결과는 originating continuation channel로 exactly once 전달한다.
  5. checkpoint writer와 compaction reservation을 한 소유권 모델로 통합한다.

Exit: 10 Keeper mixed workload에서 direct latency·FD·terminal delivery·compaction gate 동시 통과.

10. Stop / go rules

STOP

  • event queue count가 incomplete인데 pending=0으로 표시됨
  • direct 요청이 ambient provider call 두 번 이상 뒤에서 대기
  • provider tokenizer 없이 bytes/4 등으로 batch token을 추정
  • AFK를 wall-clock idle로 추정
  • 누락 verdict, job cancellation, delivery failure가 무증거 Pending으로 남음
  • background job이 turn switch 또는 server root switch에 FD를 누수

GO

  • 10/10 executable + queue read errors 0
  • admission당 attention provider call ≤ 1
  • direct enqueue→turn-start p95 ≤ 2s
  • silent loss 0, terminal exactly once
  • batch context fingerprint cardinality 1
  • 10-lane mixed harness에서 FD·compaction·connector SLO 동시 통과
