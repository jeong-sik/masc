Executive Verdict

Dashboard is broad and real, not a mock. Core shell, Monitor, Workspace, Command, Logs, and a substantial Code IDE are implemented. The weak point is not absence of screens; it is end-to-end truth wiring and operator-grade IA discipline.

Initial P0 issues were surface readiness parity, selected-repo `.masc-ide` metadata pass-through, region writer/reader storage mismatch, H2 dashboard config mutation parity, and SubBoard auth tool names. The local implementation pass fixes those wiring/auth items and carries line-level decision choice/reason/context into the IDE trace layer.

What Is Complete

  • Preact/Vite dashboard shell with route canonicalization and lazy surface dispatch.
  • Main IA surfaces: Overview, Monitor, Command, Connectors, Workspace, Lab, Code, Logs.
  • Workspace board/planning/repository/verification surfaces are largely wired.
  • Code IDE has repo tree, file content, git diff/blame, presence, annotation rendering, and right-rail context.
  • Server exposes dashboard shell, keeper, repository, credential, board, verification, transport, multimodal, and IDE routes.

What Is Missing Or Weak

  • Fixed locally: `scripts/check-dashboard-surface-parity.sh` now passes for promoted Monitor surfaces.
  • Fixed locally: Code IDE annotation/region APIs now preserve selected `repo_id`.
  • Fixed locally: generated regions append to the fixed `.masc-ide/regions.jsonl` read by HTTP.
  • Fixed locally: decision choice/reason/context now flows from keeper decision JSON to IDE trace chips.
  • Some board frontend endpoints could not be matched to server routes.
  • Fixed locally: SubBoard HTTP auth uses canonical `masc_board_sub_board_*` tool names.
  • Fixed locally: H2 excuse-pattern config mutation now requires token-bound `CanAdmin` before body read.
  • Voice and typed multimodal/image artifact workflows exist in runtime/server code but are not first-class dashboard surfaces.

Product IA Decision

Treat Dashboard as a read-mostly operator console for continuous Keeper agents. First screen should answer: who is acting, on what repo/task, with which risk, what needs human action, and where proof lives.

Do not promote every diagnostic to top-level IA. Keep operator story first; move config/doctor/transport/feature internals behind diagnostics unless parity and usage justify them.
