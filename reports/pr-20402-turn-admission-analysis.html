4. 현재 로직

최신 head 기준 turn 실행 순서다.

1. Observe board events, world observation, scheduling decision
2. Resource gates Keeper_fd_pressure.admit_turn, Keeper_disk_pressure.admit_turn
3. Record holder Keeper_turn_holders.with_recorded_turn_holder
4. Run cycle run_keeper_cycle_with_slot calls unified turn with liveness pulse
5. Cleanup Switch finalizer / protect releases holder rows
명명상 남은 문제: run_keeper_cycle_with_slot는 더 이상 slot control이 아니다. 실제 역할은 "recorded holder 아래에서 unified turn 실행 + liveness pulse/error triage"다. 다음 rename 후보는 run_keeper_cycle_with_holder 또는 run_admitted_keeper_cycle보다 더 정확한 run_keeper_cycle_with_liveness_pulse 쪽이다.
현재 pool-like controls: Keeper_registry_spawn_slots는 fleet-level max_active_keepers spawn guard이고, Keeper_fd_pressure/Keeper_disk_pressure는 resource admission guard다. keeper.turn.slot_pool_size는 deterministic pinning용 slot id 설정으로, turn capacity pool과 별개다.

7. Shell IR

남은 Shell IR 개선이 PR에 포함되었고, 현재 유일한 로컬 실패는 제거 대상 taxonomy 잔재다.

좋은 방향

  • quoted pipe는 single-stage argument로 유지된다.
  • real pipeline은 ordered stage list로 보존된다.
  • typed pipeline lowering이 raw pipeline과 같은 stage_bins를 가진다.
  • nested pipeline은 Unsupported_nested_pipeline으로 구분된다.

현재 실패

test_exec_shell_command_gate corpus row:

rg foo lib | sed s/a/b/

Expected command_not_allowed, received ok.

해석: command_not_allowed 자체가 제거 개념이므로 expected side가 stale이다.

수정 방향: 이 PR의 Shell IR 방향은 "typed stage를 보존하고 typed capability/risk 쪽에서 판단"으로 보인다. 따라서 Command_not_allowed / command_not_allowed를 baseline verdict와 사용자-facing concept에서 제거하고, 필요한 경우 더 구체적인 capability/risk denial reason으로 치환하는 쪽이 맞다.

14. 동시성 안전성 분석

2026-06-08 16:00 KST, main HEAD 666f051978 기준. #20438 이후 capacity gate가 전부 제거된 상태에서의 안전성 검증.

Per-Keeper 격리

안전

Single-fiber-by-construction. supervisor가 keeper당 정확히 1개 fiber를 fork(keeper_supervisor_launch.ml:120). heartbeat loop는 동기 꼬리재귀(let rec loop () = ...; loop ()). 같은 keeper가 2개 turn을 동시에 실행할 방법이 구조적으로 없음.

turn_running ref는 fiber-local. 주석에 명시: "Single-fiber by construction -- only this loop body reads/writes the ref."

Cross-Keeper 동시성

제한 없음

Keeper_turn_capacity(global 32 + per-keeper 2)와 Runtime_lane_capacity(per-provider max-concurrent)가 #20438에서 제거됨. 모든 keeper가 동시에 turn 실행 가능. OAS provider overload 가능.

안전성 의존: (1) FD/disk pressure circuit breaker (2) OAS HTTP timeout (3) operator가 keeper 수로 조절.

공유 상태 동기화

안전

  • registry: Atomic.t + CAS 루프. single-domain Eio에서 정확함.
  • budget_exhaustions: Stdlib.Mutex 보호. bump/reset이 원자적.
  • FD/disk pressure: Atomic.t + cas_monotonic_max. cooldown이 줄어들지 않음.
  • mark_turn_started/finished: CAS 루프. per-keeper single-fiber로 자체 경쟁 불가.

Fiber 취소 안전성

안전

Eio.Cancel.Cancelled 처리가 전체 체인에서 정확함:

  • 본문: re-raise (전파 보장)
  • cleanup: swallow (정리가 취소 전파 방해 안 함)
  • Fun.protect 의도적 미사용 — Fun.Finally_raised가 outer Cancelled를 덮는 것 방지
  • mark_turn_finished 멱등 설계 — 중복 호출 안전
