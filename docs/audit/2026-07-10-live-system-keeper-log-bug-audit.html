F-02 · P1 · Pending HITL approval blocks the whole Keeper lane

Confidence: High · Owner boundary: MASC Keeper admission / HITL continuation

같은 로그 기간에 keeper:<name> blocked on ... pending HITL approval(s) 경고가 4,700건 이상 발생했다(초기 grep 기준 4,721 records, event dedup 아님). 이는 라이브 바이너리의 기존 동작에 대한 관찰이다. 소스 흐름은 다음과 같다.

  • keeper_world_observation.ml의 cycle decision은 pending approval이 하나라도 있으면 새 cycle을 Approval_pending으로 막는다.
  • keeper_supervisor.ml는 이를 “silent chat stall”로 설명하고 있지만, 로그만 남기는 것으로는 외부 사용자에게 반응할 수 없다.
  • keeper_approval_queue.mli의 submit_and_await는 호출 fiber를 실제로 기다리게 한다. 별도의 submit_pending callback 경로가 있어도 admission precheck가 전체 lane을 다시 막는다.

스펙 위반: HITL은 약한 결합이어야 하며, Keeper는 승인 대기 중에도 busy 응답·무관한 reactive wake·다른 작업을 처리해야 한다. 승인에 종속된 continuation만 durable queue에 남기고, lane 전체를 pause하지 않는 typed continuation 구조가 필요하다.

현재 worktree 수정: queue entry에 lane_policy = Blocking | Nonblocking을 명시했다. 일반 Keeper tool approval은 Nonblocking으로 등록한 뒤 typed rejection을 반환하고, 명시적 submit_and_await 및 persisted lifecycle gate만 Blocking으로 분류한다. resolution wake는 independent-cycle entry에만 발행하고, blocking continuation에는 중복 turn을 만들지 않는다. keeper_world_observation, heartbeat board admission, supervisor recovery가 모두 typed blocking policy만 참조한다.

[근거] Xiaomi MiMo 공식 API 문서 2개, 2026-07-10 14:52 KST 확인, 신뢰도 High. OpenAI 문서는 choices.finish_reason에, Anthropic 문서는 stop_reason에 두 값을 열거한다. OCaml 5.4 variant type definitions와 variant patterns를 기준으로 public constructor와 모든 exhaustive match를 함께 갱신했다(2026-07-10 확인, 신뢰도 High).

현재 worktree 수정: OAS type/wire/pipeline 경계에 위 두 constructor를 추가했고, Responses incomplete parser는 자신의 wire enum에서 content_filter만 typed 처리한다. repetition_truncation은 on_stop을 거친 Complete로 보존하지만 임의 Unknown은 UnrecognizedStopReason으로 남는다. focused build와 Types 61개, stop-reason SSOT 4개, OpenAI streaming 40개, pipeline 50개 테스트가 성공했다. full suite/OCaml 5.4·5.5 proof는 PR CI authority로 남고, MASC live binary에는 아직 반영되지 않았다.

  • verifier pending schedule: 현재 durable meta가 명시적으로 operator_paused이므로 queue replay 손실로 판정하지 않았다. 다만 autoboot enabled와 장시간 operator pause가 충돌하고, 사용자 스펙상 pause는 정말 망가진 경우에만 허용되어야 하므로 운영 정책을 정리해야 한다.
  • deterministic retry/identical-tool guard: idle_turns=3, 동일 tool 반복 억제, 3회 실패 경고는 현재 무한루프를 제한하는 관측 가능한 안전장치다. 그 자체를 결함으로 세지 않았다.
  • legacy sandbox_profile parse reports: Keeper chat의 과거 기록에는 오류가 있으나 현재 config parse error는 0이다. 현재 장애로 재분류하지 않았다.
  • provider capacity/backpressure: 외부 capacity 오류는 관측되지만, 이를 MASC 내부 로직 결함으로 단정하지 않았다. 다만 해당 call이 다른 lane을 막지 않는지는 F-04/F-05와 함께 검증해야 한다.

현재 remediation 상태

F-01 1차 수정 완료 (worktree only): unified prompt 경로가 structured world-state user message를 더 이상 token scanner/registry strip에 넣지 않고, system/continuity instruction surfaces만 검사하도록 분리했다. keeper_unified_prompt.ml와 verification-surface regression test에 변경이 있다.

F-02 1차 수정 완료 (worktree only): keeper_approval_queue.ml에 typed blocking/nonblocking queue boundary를 추가하고, governance_pipeline.ml과 keeper_agent_run.ml의 일반 Keeper tool approval을 nonblocking 경로로 연결했다. 명시적 critical continuation gate의 blocking policy는 유지한다.

F-03 1차 수정 구현: OAS stop-reason SSOT에 공식 ContentFilter/RepetitionTruncation constructor를 추가하고 pipeline은 두 값만 typed terminal result로 보존한다. 임의 Unknown과 tool block 없는 UnmatchedToolCalls은 fail-closed를 유지한다. commit 7529dfbfd를 OAS draft PR #2506에 게시했다. focused build와 61/4/40/50 테스트가 성공했고 current-head full proof는 PR CI에서 확인한다.

OAS Types 61개, stop-reason SSOT 4개, OpenAI streaming 40개, pipeline 50개 테스트도 성공했다. 로컬 narrow build의 compiler는 ocamlc 5.5.0였고, OCaml 5.4 공식 manual을 설계 기준으로 삼았으므로 5.4 CI proof는 별도다. Dune full build/CI는 PR에서 수행한다. 테스트 실행 시 packaged model catalog 경고가 있었지만 worktree-only test fixture의 catalog discovery 경고이며, 라이브 health의 startup config warning과는 별개다.

현재 live executable은 이 worktree 변경을 포함하지 않으므로 live warning 감소나 runtime 개선으로 해석하면 안 된다. 배포/재시작 및 CI green 전에는 remediation을 완료 상태로 주장하지 않는다.
