## The drift (sub-class 2: code carries an annotated correspondence the spec didn't pick up; comment lags)

The OCaml side has *more* spec correspondence than the spec preamble advertised:

1. **Reverse-citation block in `run_keeper_cycle`.** `keeper_unified_turn.ml`'s `run_keeper_cycle` opens with a "Spec navigation (OCaml -> TLA+)" comment that names `specs/keeper-state-machine/KeeperTaskAcquisition.tla` (Cycle 8 / Tier B2, PR #11412) and maps every action: `SubmitTask` → external producers populate `observation.pending_*`; `AssignTask` → the `"turn"` channel decision (`pending_mentions <> [] || pending_board_events <> [] || pending_scope_messages <> []`); `EmptyQueueSleep` → `"scheduled_autonomous"`; `TurnComplete` → `run_turn` returns a result; `TaskRejected` → bug action, claimed task dropped without a finish. The spec preamble cited `run_keeper_cycle` but not this block — so a reader of the spec couldn't find the authoritative OCaml action mapping. **Fix**: the preamble now says `run_keeper_cycle` carries the reverse-direction citation and names the `AssignTask` → `"turn"` guard.

2. **`post_submit_task` `[@@fsm_guard]`.** `keeper_keepalive_signal.ml` has `post_submit_task ~meta ~task_id` with `[@@fsm_guard "meta.Keeper_types.current_task_id = Some task_id"]` (Cycle 44), routed through `Keeper_fsm_guard_runtime.wrap_unit ~stage:"guard"` by the PPX — a runtime post-action check mirroring "after the directive attaches the `task_id`, the meta carries it". It's invoked from `keeper_keepalive.ml`'s `assign_keeper_task_from_directive` (the operator `claim:<task_id>` path), right after `persist_directive_meta_update`. The spec preamble didn't mention it. **Fix**: the preamble now names `post_submit_task`, its `[@@fsm_guard]`, and the call site.

3. **Two producer paths, one disclosed.** `SubmitTask` in the spec is just `queue_depth' = queue_depth + 1`. In the runtime there are *two* ways a task enters: (a) the backlog producer (supervisor/autoresearch/board → `Coord.read_backlog` → next `world_observation.pending_*`), and (b) the operator `claim:<task_id>` directive (`assign_keeper_task_from_directive` — bypasses the backlog, writes `current_task_id` directly, wakes the keeper). Path (b) is a *fused `SubmitTask`+`AssignTask`* in spec terms — it both introduces the task and attaches it to the keeper — and the `post_submit_task` `[@@fsm_guard]` enforces the `current_task_id` half. The spec's `SubmitTask` abstracts the common queue/turn-state/counter effect of both, but the preamble said only "producers append tasks to the backlog". **Fix**: a new "Producer paths" paragraph in the header spells out (a) vs (b) and why the spec doesn't track `current_task_id`; the `SubmitTask` action comment now points at it.

**Fix (comment-only)**: rewrote the preamble's first paragraph to name `run_keeper_cycle`'s reverse-citation block (with the `AssignTask` guard) + `post_submit_task`'s `[@@fsm_guard]` + the `assign_keeper_task_from_directive` call site; added a "Producer paths" paragraph distinguishing the backlog producer from the operator-directive path; updated the `SubmitTask` action comment to reference it; added an iter-91 re-verification date. Model body byte-identical.
