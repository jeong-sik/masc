Keeper Boundary Runtime Audit

Sandbox 설정, Tool Surface 가시성, Model-Provider Runtime 관측성을 코드 경로와 live evidence로 대조한 보고서 및 실행 계획서.

Executive Verdict

Sandbox

NOT GREEN

코드는 TOML 기반 sandbox overlay를 갖고 있고 docker route 테스트도 있다. 그러나 live surface는 아직 서로 다르다. `analyst.toml`은 docker인데 `masc_keeper_sandbox_status`는 local로 보이고, 직전 execution receipt는 docker로 기록됐다.

Tool Surface

PARTIAL

Keeper는 도구를 실제로 보고 실행한다. Echo/Analyst 모두 allowed tools 39개, 최근 receipt는 `satisfied_execution`이다. Historical coverage gap이 현재 health를 계속 붙잡는 판정 버그는 active-gap 기준으로 패치됐다. live runtime 반영 후 `tool_call_io`, `trajectory_tool_call`, `execution_receipt`는 최신 성공 row가 과거 gap보다 새로우면 `ok`로 회복되어야 한다.

Model Runtime

NOT VERIFIED

Keeper status는 `runtime_id=ollama_cloud.deepseek-v4-pro`를 들고 있지만 `selected_model=null`, `attempt_count=0`, `runtime_outcome=not_observed`다. MASC가 실제 provider/model 실행을 검증했다고 말할 수 없다.

최종 판정: 운영은 재기동과 일부 self-repair가 계속 일어나는 중이지만, 현재 상태는 “잘 되고 있음”이 아니라 “불안정하게 살아 있고 증거 surface가 갈라져 있음”이다.

Code-Level Trace

1. Sandbox Authority Path

The intended path is:

TOML keeper profile
  -> Keeper_types_profile_toml_parser
  -> Keeper_runtime.ensure_keeper_meta overlay
  -> Keeper_registry/meta
  -> Keeper_sandbox_control + agent tool backends
  -> execution receipt + dashboard/status
CodeRoleRisk
`lib/keeper/keeper_types_profile_toml_parser.ml:123-133` Rejects removed `keeper.model`; requires `keeper.runtime_id`. Config migration is sharp. During audit this blocker appeared, then config was repaired to `runtime_id`.
`lib/keeper/keeper_meta_contract.ml` and `lib/keeper/keeper_meta_json_parse.ml:166-172` Persisted meta JSON must include `tool_access` as a canonical string array. Meta files without `tool_access` hard-fail. Tests now cover defaults for fixtures, but production decode still rejects missing field.
`lib/keeper/keeper_runtime.ml:140-152`, `580-604` Runtime reconcile should use the same declarative runtime source and update registry meta. If status reads stale registry meta, TOML docker can still appear as live local.
`test/test_keeper_sandbox_docker_route.ml:191-207`, `2290-2318` Tests now prove fixture defaults and docker routing for Execute/Grep/Read paths. Good coverage exists, but it does not prove the live status surfaces are converged.
Finding S1: Sandbox truth is split. `analyst.toml` declares docker, the latest receipt records docker, but `masc_keeper_sandbox_status` and top-level `masc_keeper_status` report local. This is the exact “Keeper does not know Sandbox” failure mode.

2. Tool Execution Path

The current audit path is the tool call log plus hook/runtime events. The old dashboard projection that rebuilt allowed/requested/materialized tool summaries was removed.

3. Model-Provider Runtime Path

The intended path is:

keeper.runtime_id in TOML
  -> Keeper_meta_contract.runtime_id_of_meta
  -> Keeper_agent_run
  -> Keeper_turn_driver.run_named
  -> Runtime.get_runtime_by_id + provider config
  -> Agent/OAS response observation
  -> receipt runtime.selected_model + model metrics
CodeRoleRisk
`lib/keeper/keeper_meta_contract.ml:612-630` Per-keeper runtime id resolves from profile defaults, then global default. Correct direction: no silent default substitution.
`lib/keeper/keeper_turn_driver.ml:99-132` Trims requested runtime id, fails fast if missing, builds runtime candidate. Good control-plane behavior; does not by itself prove provider/model execution.
`lib/keeper/keeper_agent_run_receipt.ml:146-205` Receipt `runtime_selected_model` comes from runtime observation. If observation is absent, receipt reports `selected_model=null` and attempts 0.
`lib/dashboard_provider_runs.ml:361-377`, `lib/model_inference_metrics_json.ml:140-147` Public dashboard redacts provider/model into runtime lanes and may report model_count 0. Good privacy boundary, but weak operator proof unless paired with a verified/raw internal evidence link.
Finding R1: Keeper does not currently know the actual provider/model runtime. Status shows `selected_model=null`, `attempt_count=0`, `runtime_outcome=not_observed`, and “Provider/model identity is owned by OAS.”

Execution Plan

  1. Stabilize commit truth. Bring root checkout to current `origin/main`, rebuild/restart once, and require `/health?full=1` to report the same commit as `git rev-parse origin/main`. Do not evaluate Keeper behavior while runtime is warming or flapping.
  2. Repair fleet boot first. After restart, wait one startup window. If `keeper_fibers=0`, inspect supervisor startup logs and meta decode errors. Acceptance: `keeper_fleet_safety.status` is `ok` or a bounded degraded state with named keepers, not `no_executable_keeper_fibers`.
  3. Converge sandbox truth. Make `masc_keeper_sandbox_status`, `masc_keeper_status.policy`, `runtime_trust.runtime_contract`, and receipt sandbox summary read the same overlay source. Acceptance: Analyst TOML docker yields status docker and receipt docker, or an explicit local override is shown with source.
  4. Lock tool surface evidence. Treat descriptor route evidence as canonical and make coverage gap health active-only. Acceptance: `tool_usage`, `tool_call_io`, `trajectory_tool_call`, and `execution_receipt` are `ok` or carry `active_coverage_gap_count > 0` with a fresh, actionable error for the failing source.
  5. Make runtime identity observable. Add internal-safe provider/model attribution to runtime observation and receipt. Public dashboards may keep redacted runtime lanes, but Keeper status must say either `verified=true` with source or a concrete provider blocker. Acceptance: one controlled turn produces non-null runtime observation or a named failure class.
  6. Clean persona/fleet drift. Remove stale active goal IDs, restore missing registry entries, decide whether `tech_glutton` should stay paused, and create or remove missing `base` runtime meta. Acceptance: `masc_keeper_persona_audit(include_ok=false)` returns only intentional issues.
  7. Clean docs/routes. Update stale capacity/dashboard route docs after the live route sweep. Acceptance: docs do not point operators to 404/410 endpoints for runtime capacity or dashboard batch state.
